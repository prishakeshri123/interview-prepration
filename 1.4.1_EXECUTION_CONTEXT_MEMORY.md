# 1.4.1 EXECUTION CONTEXT & MEMORY
## Complete Interview Reference Guide

**Difficulty Level:** Foundation (Asked in 100% of interviews)

**Interview Probability:** 95%+
**Real-World Usage:** Every single line of JavaScript code

## QUICK REFERENCE (5 Min Review)

### The 3 Execution Contexts
1. **Global** - Created when script starts (1 per page)
2. **Function** - Created when function called (many can exist)
3. **Eval** - Created by eval() (DON'T USE)

### Key Numbers to Remember
- **Call Stack Size:** ~10,000-100,000 frames (varies by browser)
- **Stack Overflow:** RangeError: Maximum call stack size exceeded
- **Heap Size:** Varies, typically gigabytes on modern devices
- **GC Pause:** 5-50ms on mobile (major performance concern)

### Stack vs Heap Summary
| Property | Stack | Heap |
|----------|-------|------|
| Structure | Organized, LIFO | Unorganized, chaotic |
| Speed | Fast (O(1)) | Slower (requires search) |
| Primitives | Stored directly | N/A |
| Objects | Store reference only | Store actual data |
| Lifecycle | Automatic (LIFO) | Garbage collected |
| Size Limit | Fixed, small | Large, dynamic |

## DEEP DIVE CONCEPTS

### 1. What IS an Execution Context?

**Definition:** An abstract environment where code executes. Contains everything needed to run code in that scope.

**Mental Model:** Think of it as a room or container.

```javascript
┌─ EXECUTION CONTEXT ─────────────────────┐
│                                         │
│  Variables & Functions (Scope)          │
│  this binding                           │
│  Reference to parent scope              │
│                                         │
└─────────────────────────────────────────┘
```

**What's Inside:**

```javascript
1. Variable Environment (Scope)
   - All variables declared in this context
   - let, const, var
   - Function declarations

2. This Binding
   - What 'this' refers to
   - Different in different contexts
   - Can be set by call(), apply(), bind()

3. Outer Environment Reference
   - Pointer to parent scope
   - Used for scope chain lookup
   - Null for global context
```

**How It's Created:**

```javascript
// GLOBAL CONTEXT - created automatically
var globalVar = 'exists everywhere';

// FUNCTION CONTEXT - created when function called
function example() {
  // New execution context created here
  var localVar = 'only here';
  // Context destroyed when function returns
}

// NOT a new context, same as parent
if (true) {
  var x = 5; // Not block-scoped with var!
}
```

### 2. The Call Stack (How Contexts Stack)

**Definition:** Data structure tracking which function is currently executing.

**Visualization:**

```
EMPTY (script starts)
[GLOBAL]

function a() called
[GLOBAL, a]

b() called inside a
[GLOBAL, a, b]

c() called inside b
[GLOBAL, a, b, c]

c() returns
[GLOBAL, a, b]

b() returns
[GLOBAL, a]

a() returns
[GLOBAL]
```

**Real Code with Timeline:**

```javascript
function first() {
  console.log('1');  // Stack: [GLOBAL, first]
  second();
  console.log('4');  // Stack: [GLOBAL, first]
}

function second() {
  console.log('2');  // Stack: [GLOBAL, first, second]
  third();
  console.log('3');  // Stack: [GLOBAL, first, second]
}

function third() {
  console.log('3');  // Stack: [GLOBAL, first, second, third]
}

first();

// Output:
// 1
// 2
// 3
// 3
// 4
```

**Interview Question:** Walk through call stack as code runs. Practice drawing it!

### 3. Memory: Stack vs Heap

**The Stack:**
- Where execution contexts live
- Stores primitive values directly
- Organized, automatic cleanup
- Finite size (overflow error)

```javascript
function example() {
  var x = 5;           // Stored in stack
  var y = "hello";     // Stored in stack
  var obj = { a: 1 }; // Reference stored in stack
                       // Object stored in heap
}
// When example() returns, entire context removed from stack
// x, y, obj references all disappear
```

**The Heap:**
- Where objects, arrays, and complex data live
- Unstructured, requires search
- Garbage collected (automatic, but imperfect)
- Much larger than stack

```javascript
var bigObject = {
  data: new Array(1000000),
  nested: {
    more: new Array(1000000)
  }
};
// bigObject reference = 8 bytes in stack
// Actual object = megabytes in heap

bigObject = null; // Reference removed
// GC sees nothing points to the object anymore
// Eventually deletes it from heap (timing varies)
```

**Primitive vs Reference Types:**

```javascript
// PRIMITIVES: Stored in stack, copied by value
let x = 5;
let y = x;
y = 10;
console.log(x); // 5 (unchanged)

// REFERENCES: Pointer stored in stack, object in heap, copied by reference
let obj1 = { name: 'Alice' };
let obj2 = obj1; // Both point to same object in heap
obj2.name = 'Bob';
console.log(obj1.name); // 'Bob' (changed!)
```

### 4. Garbage Collection (How Memory Gets Freed)

**Why It Matters:** Memory leaks crash apps. GC pauses cause jank. Understanding GC is critical for scale.

#### Mark & Sweep Algorithm (Modern)

**Step 1: Mark Phase**
```
Starting from global context (the root):
1. Find all reachable objects
2. Mark them as "alive"
3. Anything not marked is "dead"
```

**Step 2: Sweep Phase**
```
Delete all unmarked objects from heap
Reclaim their memory
```

**Visual:**

```
Before GC:
Reachable:  ✓ obj1, ✓ arr1, ✓ func1
Unreachable: ✗ orphanedData, ✗ oldArray

After GC:
Reachable:  ✓ obj1, ✓ arr1, ✓ func1
Unreachable: [DELETED]
Memory freed!
```

**Code Example:**

```javascript
function createLargeObject() {
  let hugeArray = new Array(1000000).fill('data');
  let orphanedData = new Array(1000000).fill('x');
  
  return () => {
    console.log(hugeArray.length);  // Closure captures hugeArray
    // orphanedData is not captured
  };
}

const fn = createLargeObject();

// When createLargeObject() returns:
// hugeArray: REACHABLE (closure captures it)
// orphanedData: UNREACHABLE (nothing references it)
// GC deletes orphanedData from heap
```

#### When Objects DON'T Get Deleted

**Problem 1: Forgotten Event Listeners**
```javascript
// ❌ LEAKS
const handler = () => console.log(data); // Captures data
element.addEventListener('click', handler);
// Never called: element.removeEventListener('click', handler);

// Result: handler's closure keeps 'data' in memory forever
// Even if data is huge, it stays alive
```

**Problem 2: Global Variables**
```javascript
window.cache = new Array(1000000); // Stored globally
// NEVER garbage collected (globals are always reachable from root)
// This is permanent memory usage
```

**Problem 3: Timers**
```javascript
setInterval(() => {
  processData(bigObject); // Closure captures bigObject
}, 1000);

// Even if you don't need bigObject anymore:
// - Timer keeps running
// - Closure keeps bigObject alive
// - Memory never freed
```

**Problem 4: Circular References (Legacy Issue)**
```javascript
// With reference counting (old GC), this leaks:
let a = { ref: null };
let b = { ref: null };
a.ref = b;
b.ref = a; // Circular reference!
a = null;
b = null;
// Old GC: counts never reach 0, both leak
// Modern GC (Mark & Sweep): recognizes unreachable cycle, deletes both

// Modern JavaScript: Not a problem
// But good to know why reference counting failed
```

### 5. Scope Chain (How Variables Are Found)

**Definition:** The path JavaScript follows to find a variable.

**The Lookup Order:**

```
1. Local scope (current execution context)
2. Parent function's scope
3. Parent's parent's scope
...
4. Global scope
5. Not found? → ReferenceError
```

**Code Example:**

```javascript
const global = 'global';

function outer() {
  const outerVar = 'outer';
  
  function inner() {
    const innerVar = 'inner';
    
    // Scope chain: inner → outer → global
    
    console.log(innerVar);  // Found in inner (step 1)
    console.log(outerVar);  // Not in inner, found in outer (step 2)
    console.log(global);    // Not in inner/outer, found in global (step 3)
  }
  
  inner();
}

outer();
```

**Performance Implication:**

```javascript
// BAD: Deep scope chain
function a() {
  function b() {
    function c() {
      function d() {
        function e() {
          return x; // Looks up 5 levels!
        }
      }
    }
  }
}

// GOOD: Shallow scope chain
let x = 10;
function simple() {
  return x; // Looks up 1 level (fast)
}
```

### 6. Closures (The Most Important Pattern)

**Definition:** NOT a function. The phenomenon where a function retains access to outer scope variables even after the outer function returns.

**Formula:**
```
Closure = Function + Captured Scope
```

**Classic Example:**

```javascript
function makeCounter() {
  let count = 0;
  
  return function() {
    return ++count;  // Captured 'count' variable
  };
}

const counter = makeCounter();
console.log(counter()); // 1
console.log(counter()); // 2
console.log(counter()); // 3

// How it works:
// 1. makeCounter() executes, creates 'count'
// 2. Returns inner function
// 3. makeCounter() context REMOVED from stack
// 4. But 'count' is NOT deleted because closure references it
// 5. Each call to counter() finds 'count' in closure
```

**Why It Works:**

```
Inner function's [[Scope]]:
[
  { count: 0 },              // Inner's scope
  { makeCounter's scope },   // Captured scope (closure)
  { global scope }           // Global
]

GC Analysis:
- counter references inner function
- Inner function references count
- count is reachable
- count cannot be deleted
- Persists forever
```

**Practical Example:**

```javascript
// Data privacy
function createUser(name) {
  let _secret = 'confidential';
  
  return {
    getName: () => name,
    getSecret: () => _secret, // Only way to access _secret
    setSecret: (s) => _secret = s
  };
}

const user = createUser('Alice');
console.log(user.getName());        // 'Alice'
console.log(user._secret);          // undefined (private!)
console.log(user.getSecret());      // 'confidential' (through method)
```

## INTERVIEW QUESTIONS & ANSWERS

### Easy Q1: Explain Execution Context
**What They Want:**
Basic understanding of the concept and three types.

**Perfect Answer:**
"An execution context is an environment where code runs. It contains a scope (variables), this binding, and a reference to the outer scope. There are three types: global (created once when script loads), function (created when function called), and eval (don't use). When a function returns, its context is popped from the call stack and its variables are eligible for garbage collection."

**Score Yourself:**
- ✓ Mentioned all three types (5 pts)
- ✓ Explained what goes in a context (5 pts)
- ✓ Mentioned call stack and GC (5 pts)

---

### Easy Q2: Stack vs Heap
**What They Want:**
Clear distinction and why it matters.

**Perfect Answer:**
"The stack stores execution contexts and primitive values. It's organized and fast, but has a size limit. When a function returns, its context is removed from the stack.

The heap stores objects, arrays, and complex data. When a variable references an object, the stack stores a reference (memory address) to the heap.

This means primitives are copied by value (let y = x copies the value), but objects are copied by reference (let obj2 = obj1 makes both point to the same object in memory).

This is why changing obj2 affects obj1 but changing y doesn't affect x."

**Score Yourself:**
- ✓ Explained stack structure and primitives (3 pts)
- ✓ Explained heap and references (3 pts)
- ✓ Connected to copy-by-value vs copy-by-reference (4 pts)

---

### Medium Q4: Explain Closures
**What They Want:**
Deep understanding, not just definition.

**Perfect Answer:**
"A closure is not a function; it's the phenomenon where a function retains access to outer scope variables even after the outer function returns.

In the makeCounter example, the inner function captures the 'count' variable. When makeCounter() finishes and its execution context is popped from the stack, 'count' is NOT garbage collected because the inner function's [[Scope]] chain still references it.

Each time we call counter(), a new execution context is created for the inner function. It looks up 'count' in its scope chain, increments it, and returns the value. The 'count' variable persists across all calls because it's captured by the closure.

This works because the garbage collector sees that the inner function is alive and points to count, so count must stay alive."

**Score Yourself:**
- ✓ Explained what closure is (not just the function) (3 pts)
- ✓ Explained scope chain (3 pts)
- ✓ Explained GC reasoning (3 pts)
- ✓ Mentioned how data persists (2 pts)

---

### Hard Q7: Garbage Collection
**What They Want:**
Mark & Sweep explanation and when objects leak.

**Perfect Answer:**
"Mark-and-sweep has two phases. In the mark phase, starting from the global context (the root), the GC walks the reference graph and marks every reachable object. Anything not marked is unreachable.

In the sweep phase, the GC deletes all unmarked objects from memory.

Objects don't get deleted when:
1. Event listeners keep closures alive (forgotten removeEventListener)
2. Timers have closures capturing data
3. Global variables (always reachable from root)
4. Circular references (modern GC handles this, but shows why reference counting failed)

The key insight is that GC can only delete what's unreachable. If any chain of references reaches an object from the root (global context), it stays alive."

**Score Yourself:**
- ✓ Explained mark & sweep (4 pts)
- ✓ Mentioned multiple leak scenarios (4 pts)
- ✓ Explained why each is a leak (2 pts)

## PRODUCTION PROBLEMS & SOLUTIONS

### Problem 1: React Memory Leak

**The Leak:**
```javascript
useEffect(() => {
  const subscription = api.subscribe(callback);
  // Missing cleanup!
}, []);
```

**Why It Leaks:**
- Callback captures component state through closure
- Subscription stays alive forever
- Component can't be garbage collected
- After 100 page visits: 100 leaked components

**The Fix:**
```javascript
useEffect(() => {
  const subscription = api.subscribe(callback);
  return () => subscription.unsubscribe(); // ✅ Cleanup
}, []);
```

**Interview Impact:** Shows you understand closures + React lifecycle + memory management.

---

### Problem 2: Var Scope Trap

**The Leak:**
```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i), 0);
}
// Output: 3, 3, 3 (not 0, 1, 2!)
```

**Why It Fails:**
- `var i` is function-scoped, not block-scoped
- All iterations share the SAME i variable
- By loop end, i = 3
- All callbacks log 3

**The Fix:**
```javascript
for (let i = 0; i < 3; i++) {  // let = block-scoped
  setTimeout(() => console.log(i), 0);
}
// Output: 0, 1, 2 ✓
// Each iteration gets its own i variable
```

---

### Problem 3: Stack Overflow

**The Problem:**
```javascript
function recursive() {
  const data = new Array(1000000);
  recursive(); // Infinite recursion
}
recursive();
// RangeError: Maximum call stack size exceeded
```

**Why It Crashes:**
- Each call adds frame to stack
- Stack has finite size
- Stack overflows before heap is exhausted
- Not about memory, about stack structure

**The Fix:**
```javascript
// Convert to iterative approach
function processIterative(n) {
  const stack = [n]; // Use array as explicit stack
  while (stack.length > 0) {
    const item = stack.pop();
    // Process item
  }
}

// Or use trampolining
function trampoline(fn) {
  let result = fn();
  while (typeof result === 'function') {
    result = result();
  }
  return result;
}
```

### Exercise 2: Spot the Memory Leak
Code:
```javascript
class Component {
  constructor(element) {
    this.element = element;
    this.data = new Array(1000000);
    
    this.element.addEventListener('click', () => {
      console.log(this.data.length);
    });
  }
}

const comp = new Component(document.getElementById('btn'));
// User navigates away
comp = null; // Try to free memory
```

**Your Task:** Where's the leak?

<details>
<summary>Solution</summary>

The event listener's callback captures 'this' (the Component instance). Even though comp = null, the listener keeps the Component alive in memory.

Fix: Add cleanup
```javascript
destroy() {
  this.element.removeEventListener('click', this.handler);
  this.element = null;
  this.data = null;
}
```

</details>

## GOTCHAS & TRAPS

### Gotcha 1: Closure Captures Reference, Not Value
```javascript
let funcs = [];
for (let i = 0; i < 3; i++) {
  funcs.push(() => i);
}
funcs[0](); // 0 ✓ Correct with let
funcs[1](); // 1 ✓ Each iteration has own i

// With var:
let funcs = [];
for (var i = 0; i < 3; i++) {
  funcs.push(() => i);
}
funcs[0](); // 3 ✗ All reference same i
funcs[1](); // 3 ✗ Which is now 3
```

### Gotcha 2: === Compares References, Not Values
```javascript
let obj1 = { x: 5 };
let obj2 = { x: 5 };
obj1 === obj2; // false (different references)

let obj3 = obj1;
obj3 === obj1; // true (same reference)
```

### Gotcha 3: Global Variables Never Get GC'd
```javascript
window.cache = new Array(1000000); // Permanent!
window.cache = null; // Still takes memory until page reload
// Globals are always reachable from root
```

## MEASUREMENT & DEBUGGING

### How to Detect Memory Leaks

**Method 1: Chrome DevTools Memory Profiler**
1. Open DevTools → Memory tab
2. Take heap snapshot (baseline)
3. Do action that might leak (add items, navigate, etc.)
4. Take another snapshot
5. Compare: heap should shrink back if no leak

**Method 2: Monitor Heap Size**
```javascript
if (performance.memory) {
  setInterval(() => {
    console.log('Heap:', performance.memory.usedJSHeapSize);
  }, 1000);
}
```

**Method 3: Check Detached DOM Nodes**
DevTools → Memory → Take snapshot → Filter: "detached"

## KEY TAKEAWAYS

1. **Execution contexts are created/destroyed constantly** - understand when and why
2. **Stack stores references, heap stores data** - this affects copying behavior
3. **Closures are powerful but dangerous** - they prevent garbage collection
4. **Event listeners are the #1 cause of leaks** - always cleanup
5. **Var is function-scoped, let/const are block-scoped** - this matters for loops
6. **GC is imperfect** - you must help it by breaking references
7. **Stack has size limit, heap is large** - overflow is about stack, not memory
